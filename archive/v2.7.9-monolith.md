# prompt-training v2.7.9

> **Track note**: This document consolidates the t1/t2 (Token Budget Experiment) track back into the main track. v2.7.9 = v2.7.8.t2.5 + v2.7.8.t2.6 (GAIA integration + verifier + operational spec), merged into a single implementation document. The 80/70/150 token split introduced in the t track is now considered stable for v0 and may still be re-tuned based on results.

> "If an AI is going to do research well, it must first refine the problem."

When you give a vague prompt directly to an agent, it falls into infinite loops a lot of the time. If you first run the prompt through an inverse model to lower its semantic entropy, the loop rate drops sharply. This project tests that idea.

## Core hypothesis

> Compare two conditions: (A) give a vague prompt directly to the agent, and (B) first run it through an inverse model to lower semantic entropy, then give it to the agent. In condition B, the agent falls into infinite loops far less often.

## Theoretical background

Borrowed from the motor neuroscience model of Wolpert & Kawato (1998). The brain uses forward models (predict) and inverse models (work backward) as a pair.

**Inverse model**: desired state → work backward to find the actions needed.

| Step | Content | Description |
|------|---------|-------------|
| Input | prompt (vague intent) | Take in the raw prompt |
| Target | Define the perfect "done" state | "When this is finished, what exists?" |
| Invert | Work backward | "To reach that, what must be done and known?" |
| Output | Improved prompt | Compress the result into one clear instruction |

In v0, only the inverse model is tested. The forward model (verification during execution) will be considered in v1, after we see the results from condition B.

## Why fewer loops in B

Condition B has fewer loops because the entropy (H) at the moment the agent starts running is already low enough. In condition A, loops are detected in the region where **d²H/dt² ≈ 0 while H is still high**. In condition B, by the time d²H/dt² ≈ 0, H is already in the convergence region, so this is read as completion, not as a loop.

## Measuring semantic entropy

### What we measure: H_raw vs H_improved (2 points)

Every entropy measurement uses the same fixed **minimal instruction** as the system prompt. This is a control: it makes sure any difference in the measured value comes only from the query/context, not from changes in the system prompt.

**Minimal instruction**: `"You are an AI agent that solves problems using tools."`

We use the same measurement question ("What concrete action will the agent take next?") on the two inputs below:

- **H_raw**: `[minimal instruction] + [original query summarized to 80 tokens]` → "Next action?" × N times → entropy
- **H_improved**: `[minimal instruction] + [refined query summarized to 80 tokens]` → same question × N times → entropy
- **ΔH = H_raw − H_improved**: the effect size of pre-processing

**Apples-to-apples**: Both measurements go through the same 80-token summarization pipeline. The refined query is likely to exceed 80 tokens because it carries detailed instructions, so without compressing both with the same yardstick, we couldn't tell whether ΔH comes from "a difference in semantic clarity" or "a difference in context length." Also, the inputs used to measure H_raw/H_improved must match exactly the text that goes into the Head during execution; otherwise comparison with H_n is not valid.

Because the system prompt, the measurement question, and the summarization pipeline are all fixed, the only thing that can produce a difference between the two values is the semantic clarity of the query.

We do not measure entropy at the intermediate steps (Target, Invert). Each step asks a different question, so the denominator changes and the values can't be placed on the same curve. The question for v0 is "does it work," not "why does it work." If we need debugging information later, we add it in v1.

### How we measure

At each measurement point we sample the same prompt/state N times (default 10), embed the responses with a neutral embedding model (Llama-3 via Together AI), build semantic clusters using cosine-similarity hierarchical clustering (AgglomerativeClustering, average linkage, distance threshold 0.15), and compute the Shannon entropy of the cluster distribution.

```
H = −Σ p(c) × log₂(p(c))
```

- It's like asking 10 people "What's the next action?" in the same situation.
- If all answers differ → H is high (vague).
- If all answers are the same → H ≈ 0 (clear).

### Handling measurement noise: let summarization variation cancel out in ΔH

**Problem: if you measure the same task twice, the H value comes out a little different each time.**

The reason is the summarization step. The 80-token summary is generated fresh by an LLM each time, and the result is not always the same. For example, if the original query is "I want to raise team productivity," two summaries might come out like this:

- First time: "Looking for ways to improve team productivity"
- Second time: "Searching for methods to improve team work efficiency"

To a human these mean the same thing, but when you ask "what's the next action?", the version with the word "productivity" might pull answers toward tool recommendations, while the version with "work efficiency" might pull them toward process analysis. A single word changes the response distribution, and so the H value changes.

**Decision: don't try to remove this variation. Absorb it into the H measurement.**

We re-run the summarization on every sample. In other words, we treat "summarize + respond" as a single block and measure the H of the whole pipeline. The fact that the summary is different each time becomes part of H. We do not artificially try to control this variation.

**Mechanism: the noise cancels out automatically in the ΔH difference.**

Conditions A and B use the same summarizing LLM and the same summarization pipeline (80-token compression). So the noise from the summarization step gets baked into H_raw and H_improved equally. When you compute ΔH = H_raw − H_improved, the noise that was in both sides drops out, and what remains is the signal we actually want to see — the difference in semantic clarity between the original and the refined query.

> **Tape measure analogy**: It's like measuring the height difference between two people with the same tape measure. Even if the tape is off by 1 cm, the error disappears in the difference. A − B still gives the correct height difference.

### False Positive control: separating termination reasons

**Problem: a task that runs out of `max_steps` can be misjudged as a loop.**

When a complex task is being worked through step by step and hits the step limit before finishing, the entropy curve over its last few steps may happen to satisfy `d²H/dt² ≈ 0 AND H > 0.3 × H_raw`. The actual situation is "not enough steps for the complexity," but the result table records it as "loop occurred." This is a serious source of noise when interpreting v0 results — even if condition A shows a high loop rate, we can't tell whether it's caused by real ambiguity or by running out of steps.

**Decision: record the termination reason as one of four categories.**

- `completed`: the agent produced a final answer (normal termination)
- `loop_detected`: detect_loop returned positive (a real loop)
- `max_steps_reached`: max_steps was hit but the loop condition was not satisfied
- `error`: infrastructure failure (LLM empty response, inverse() failure, tool call parsing failure) — see Failure Handling section

The loop count counts only `loop_detected`. `max_steps_reached` and `error` are separate categories and never get folded into the loop count.

**Mechanism: apply two controls together.**

1. **Separating termination reasons (mandatory)**: removes the False Positive itself from the results at no cost. No matter how `max_steps` is set, the interpretation of the results stays clean.
2. **Level-based `max_steps` assignment**: derived from GAIA Level (see Task Set section). Combined with control 1, both the reliability of the experiment and the diversity of tasks go up.

Control 1 is the primary principle. Without it, doing only control 2 raises experiment cost while leaving the reliability of the interpretation dependent on the choice of `max_steps`. With control 1 in place, `max_steps` is just a resource limit, not a variable that decides the outcome of the experiment.

### False Negative control: separating termination mechanism from result correctness

**Problem: t2.5 considered a task "completed" as long as the agent produced a final answer, regardless of whether the answer was correct.**

The t2.5 `completion rate` metric was effectively "agent terminated normally" and said nothing about whether the experiment actually succeeded. A run where every agent confidently produced a wrong answer would be indistinguishable from a run where every agent answered correctly.

**Decision: record termination mechanism and result correctness as two orthogonal columns.**

- `terminated_by`: records *how* the task ended (completed / loop_detected / max_steps_reached / error)
- `verifier_passed`: records *whether the result was correct* (True / False / N/A)

These two dimensions are orthogonal and must never be collapsed into one column. Every derived metric reads from exactly one column:

- `loop_count` ← `terminated_by == "loop_detected"`
- `correct_count` ← `terminated_by == "completed" AND verifier_passed == True`

This is the direct counterpart of the t2.5 False Positive control (separating loop from resource-limit termination). Together they form a complete termination taxonomy.

### Token management strategy (Lean & Forward Strategy)

Entropy measurement caps the context at **300 tokens**, dynamically allocated as follows to support the Forward Model's "Predict-Compare" cycle.

> **Note**: The Forward Model references in this section describe structure prepared for v1 activation. v0 does not exercise the Predict-Compare cycle directly; only the Inverse Model (pre-processing) is tested.

> **Design philosophy: summarization vs trimming**
>
> Within the 300-token limit, we need to preserve both the "direction of the goal" and the "resolution of the measurement." So we apply different compression methods to Head and Tail.
>
> - **Head (summarization / abstraction):** Reorganize the user's wordy requirements at a high level, leaving only the **final goal (Target)** the agent must not lose, like a compass.
> - **Tail (trimming / preserving resolution):** If we summarize the agent's current state (error logs, deliberation, etc.), uncertainty (entropy) gets artificially measured as low. So we don't rewrite meaning. Instead we mechanically strip out unnecessary whitespace and filler with **trimming (pruning)**, preserving the specificity and resolution of the original text.

**1. Head (Target) — 80 tokens (summarized once and fixed)**
- Composition: `[minimal instruction] + [query summarized to under 80 tokens]`
- Role: the **Compare** anchor for the Forward Model. Preserves the final goal (Definition of Done).
- Policy: not a length cut, but a meaning-level **summarization**. No matter how long the input query is, the essence of the task (Definition of Done) is compressed and locked into 80 tokens.
- **When summarized: once, before execution. Never modified at any later step. Held fixed.**

**2. Body (State) — 70 tokens (summarized)**
- Composition: a heavily compressed summary of history before step n−2
- Role: state value for the Forward Model's **Predict**. Updated to focus on "facts confirmed so far."
- Policy: re-summarized recursively at each step.

**3. Tail (Action) — 150 tokens (uniform semantic trimming)**
- Composition: the detailed Thought and tool Observation from the previous step (n−1)
- Role: the **target of verification** for the Forward Model. Preserves the specificity of actions to keep the resolution of the entropy measurement.
- Policy: To prevent distortion of the entropy measurement, we exclude any summarization that changes meaning. We keep the original wording of key error logs and technical keywords and only trim length, applying the same trimming rule uniformly across all conditions. **Every step passes through the trim function; for short inputs the original is effectively preserved.**

> ※ At every measurement point we keep the same question template — "What concrete action will the agent take next?" — to keep measurement consistent.

> **Stability note**: the 80/70/150 split inherited from the t track is considered stable for v0 and may still be re-tuned based on empirical results.

### Tavily responses and Tail trimming (clarification)

`tavily_search` returns approximately 5 results × ~200 characters per snippet, totaling ~1000 characters, which exceeds the Tail 150-token budget. `tavily_extract` can return even longer page bodies. The handling follows the Tail trimming principle above:

1. **In the current step's Thought phase**: the agent sees the full, untruncated Tavily response. No truncation at this point — the Thought's reasoning quality must not be degraded.
2. **In the next step's context construction only**: the raw Observation is passed through `trim_to_tail(obs, max_tokens=150)` before becoming the Tail. Trimming is meaning-preserving (whitespace/filler removal), not summarization.
3. **For `tavily_extract`**: the raw page body is shown to the Thought once, then trimmed for subsequent steps.

Implementation must separate the "original Observation → current Thought" path from the "original Observation → trim → next step's Tail" path.

## Experiment structure

### Condition A (baseline): [minimal instruction + 80-token-summarized original query] → ReAct agent

```
[original query] → 80-token summary (once, Head fixed)
  → [minimal instruction + summarized query] → measure H_raw
  → run ReAct agent ([minimal instruction + summarized query])
    → each step: Thought → Action → Observation
    → measure Hₙ at each step (300-token context: 80 Head + 70 Body + 150 Tail)
  → terminate as one of: completed / loop_detected / max_steps_reached / error
  → if completed: call verifier
```

### Condition B (treatment): inverse → 80-token summary → ReAct agent

```
[original query] → 80-token summary (once) → [minimal instruction + summarized original] → measure H_raw
  → run inverse(original query)
    → Target (define the done state)
    → Invert (work backward: needed actions and knowledge)
    → Output (refined query)
  → refined query → 80-token summary (once, Head fixed)
  → [minimal instruction + summarized refined] → measure H_improved
  → run ReAct agent ([minimal instruction + summarized refined])
    → each step: Thought → Action → Observation
    → measure Hₙ at each step (300-token context: 80 Head + 70 Body + 150 Tail)
  → terminate as one of: completed / loop_detected / max_steps_reached / error
  → if completed: call verifier
```

### Metrics (v0 minimal set: 3 + 3)

**Result metrics (after execution):**

| # | Metric | Definition | Why |
|---|--------|-----------|-----|
| 1 | Number of loops | count(terminated_by == "loop_detected") | Direct dependent variable of the hypothesis |
| 2 | Token usage | sum of input + output tokens across all LLM calls | Cost effect |
| 3 | Task correctness rate | count(terminated_by == "completed" AND verifier_passed == True) / total_tasks | Whether the agent actually produced correct answers |

**Process metrics (live during execution):**

| # | Metric | What it measures |
|---|--------|------------------|
| 4 | H (entropy) | Uncertainty level at each step |
| 5 | dH/dt (velocity) | Rate of entropy decrease between steps |
| 6 | d²H/dt² (acceleration) | Acceleration/deceleration of the decrease |

**Loop detection condition:** d²H/dt² ≈ 0 (within window) AND H > α × H_raw
- threshold = 0.3 × H_raw (based on the raw prompt's semantic entropy, α=0.3)
- Both A and B use H_raw as the reference. We do not switch to H_improved as the reference in B.

**Candidate metrics for v1:** completion time, TTFMA, pivot count, spec match, error recovery rate, number of human interventions.

## Task set: GAIA Level 1 (text-only)

### Source

The v0 task set is **GAIA validation Level 1, text-only subset**. Self-authored tasks are not used.

**Rationale for choosing GAIA**:
- Questions are conceptually simple for humans but require multi-step reasoning, tool use, and information retrieval — precisely the setting where agent loops emerge.
- Ground truth answers are unambiguous and machine-verifiable, enabling automated scoring.
- Level 1 is the smallest, most manageable subset for v0 while still exercising the hypothesis.

**Rationale for text-only filter**:
- v0 targets the inverse model's effect on agent loops. Multimodal processing (images, audio, PDFs) is a separate axis of capability that would confound the result.
- Multimodal is deferred to v1.

**Rationale for excluding self-authored tasks**:
- GAIA provides a pre-existing, widely recognized benchmark with an official scorer. Self-authored tasks would require a separate (and weaker) verification path.
- Korean-language tasks are deliberately excluded. GAIA's scorer is English-only, and maintaining a single scoring pipeline is more valuable than language generalization in v0. Language generalization is a v1 concern or a separate paper.

### Loader

The loader applies a **two-stage filter**:

```python
import re
from datasets import load_dataset

# Patterns for external multimodal resources that Tavily cannot handle.
# These appear in the Question body even when file_name is empty.
# Observed in GAIA Level 1: questions like "In the video https://youtube.com/watch..."
MULTIMODAL_URL_PATTERNS = [
    r"youtube\.com/watch",
    r"youtu\.be/",
    r"\.mp[34]\b",   # mp3, mp4
    r"\.wav\b",
    r"vimeo\.com/",
]

def is_truly_text_only(task) -> bool:
    """
    Text-only means BOTH:
    1. No file attachment (file_name is empty), AND
    2. Question body does not reference external multimodal resources
       that Tavily's search/extract cannot handle.
    """
    if task.get("file_name"):
        return False
    question = task.get("Question", "")
    for pattern in MULTIMODAL_URL_PATTERNS:
        if re.search(pattern, question, re.IGNORECASE):
            return False
    return True

def load_gaia_tasks():
    """
    Load GAIA validation set Level 1, truly text-only tasks.
    
    Column schema (confirmed):
        task_id, Question, Level, Final answer, file_name, file_path, Annotator Metadata
    Level 1 validation total: 53 tasks.
    """
    gaia = load_dataset("gaia-benchmark/GAIA", "2023_level1", split="validation")
    
    # Stage 1: empty file_name
    no_file = [t for t in gaia if not t.get("file_name")]
    
    # Stage 2: no multimodal URLs in Question
    text_only = [t for t in no_file if is_truly_text_only(t)]
    
    total = len(gaia)
    excluded_by_url = len(no_file) - len(text_only)
    print(f"[load_gaia_tasks] Level 1 total: {total}")
    print(f"[load_gaia_tasks] After file_name filter: {len(no_file)}")
    print(f"[load_gaia_tasks] After multimodal URL filter: {len(text_only)}")
    print(f"[load_gaia_tasks] Excluded due to multimodal URLs in question body: {excluded_by_url}")
    return text_only
```

**Why two stages**: The first stage alone is insufficient. GAIA Level 1 contains tasks where `file_name` is empty but the Question body embeds a YouTube/video/audio URL. These are structurally unsolvable with Tavily alone and would inject systematic noise into Condition A's loop rate, confounding the inverse model's effect. Logging both stage counts separately makes the filter behavior transparent — if the URL filter excludes an unexpectedly large fraction, that itself is a signal worth investigating.

### Schema mapping (GAIA field → internal task field)

| Internal field | GAIA field | Notes |
|---|---|---|
| `task_id` | `task_id` (GAIA UUID) | used as cache key and TSV row |
| `raw_prompt` | `Question` | passed to inverse() and to the agent |
| `ground_truth` | `Final answer` | passed to verifier only; never seen by agent |
| `level` | `Level` | always 1 in v0 |
| `max_steps` | derived from Level | see below |
| `timeout_seconds` | constant 300 | — |

**Fields explicitly not used**:
- `Annotator Metadata`: contains annotator's step-by-step solution procedure. Using this would leak ground-truth signal to the agent.
- `file_path`: only relevant for tasks with attachments, which are excluded.
- `expected_outputs` (from t2.5): removed entirely; replaced by `ground_truth` + verifier.
- `domain` (from t2.5): removed; replaced by `level`.

**Important**: GAIA column names contain spaces (`"Final answer"`). Dict access must be `task["Final answer"]`, not `task["final_answer"]`.

### `max_steps` by Level

| Level | max_steps |
|---|---|
| 1 | 15 |
| 2 | 25 |
| 3 | 50 |

v0 uses Level 1 only, so effectively `max_steps = 15` for all tasks. Levels 2 and 3 are listed for v1 preparation.

**Contingency rule (max_steps upward adjustment)**: If the first full run shows that more than 20% of Level 1 tasks end with `terminated_by == "max_steps_reached"`, raise `max_steps` to 20 and record the results separately. This is not a spec modification — it is a resource limit adjustment applied equally to both conditions A and B, so it is not an experimental variable.

### Sample size contingency

GAIA Level 1 validation has **53 tasks total**. After the two-stage text-only filter, the exact count must be logged at first load.

**Contingency rule (applied once at first load, then locked)**:
1. Text-only Level 1 ≥ 30: proceed as-is
2. Text-only Level 1 in [20, 29]: add Level 2 text-only tasks until the combined count reaches 30
3. Text-only Level 1 < 20: report v0 results as descriptive only (no statistical tests)

Sample composition is fixed after the first load and does not change between runs. Reruns always target the identical task set.

## Agent

### Tool set

The ReAct agent has exactly **three tools**:

| Tool | Purpose | Backend | Tavily credit |
|---|---|---|---|
| `tavily_search(query: str)` | Web search | Tavily basic mode (`max_results=5`) | 1 |
| `tavily_extract(url: str)` | Fetch and extract page text | Tavily | 1 |
| `final_answer(answer: str)` | Terminate the ReAct loop | Internal | 0 |

**Constraints and rationale**:
- **Basic mode fixed**: Tavily has basic (1 credit) and advanced (2 credits) modes. v0 uses basic only, for cost predictability and uniform search quality across conditions A and B.
- **`max_results=5` fixed**: Raising this increases Tail-trimming burden; lowering degrades search quality. Fixed at 5 (Tavily default) to remove it as an experimental variable.
- **Exactly three tools**: Adding more tools introduces "which tool to use" as a new variable that would confound the inverse model's effect. Held fixed in v0.
- **`final_answer` termination**: When the agent emits a tool_use block for `final_answer`, `run_react_loop` immediately exits with `terminated_by = "completed"`. If the agent produces only text (no tool call), the step is retried once; on second failure, the loop forces a `final_answer` call with the agent's best-effort answer.

### Final answer format

GAIA's scorer expects answers in a specific normalized form. The format instruction lives in the **`final_answer` tool description**, NOT in the minimal instruction. This preserves the apples-to-apples control variable: the entropy measurement input is still `[minimal instruction] + [80-token summary]`, identical for A and B.

```
Tool: final_answer
Description: Call this tool when you are ready to give the final answer to the user's question.

Your answer should be:
- A number, OR
- As few words as possible, OR
- A comma-separated list of numbers and/or strings.

Formatting rules:
- If you are asked for a number, do not use commas or units (such as $ or %) unless explicitly requested.
- If you are asked for a string, do not use articles or abbreviations, and write digits in plain text unless explicitly requested.
- If you are asked for a comma-separated list, apply the above rules to each element in the list.

Parameter:
    answer (str): the final answer, formatted according to the rules above.
```

The format instruction exists only in the agent's tool schema. It is not passed to the entropy measurement pipeline.

### LLM model (single-model policy)

**All LLM calls in v0 use the same model**, defined as `MODEL` at the top of `benchmark.py` and overridable via CLI `--model`.

Applies to:
- `inverse()` internal calls (Target / Invert / Output)
- `summarize_to_head()`
- ReAct agent (Thought + Action generation)
- `measure_semantic_entropy()` sampling calls

**Exception**: the embedding model for semantic clustering is Together AI's Llama-3 (decided in t2.4, maintained here). This is a separate component from the generation model and does not affect the single-model policy.

**Rationale**: v0 minimizes experimental variables. The hypothesis under test is "does inverse pre-processing reduce loops?" — introducing model asymmetry (e.g., agent=Sonnet, inverse=Opus, summarize=Haiku) would require an exponential ablation. Asymmetric model studies are deferred to v1.

## Detailed design per file

### 1. `inverse.py` — Inverse Model engine + entropy measurement

Role: take a vague prompt and return an improved prompt. Entropy is measured only at the entrance (H_raw) and the exit (H_improved).

```python
"""
prompt-training/inverse.py

Vague prompt → improved prompt converter (inverse model).
Entropy is measured only at the raw prompt and the improved prompt.

Usage:
  from inverse import inverse, measure_semantic_entropy

  result = inverse("I want to cut waste. I have 3 cards.", model=MODEL)
  print(result["improved_prompt"])
  print(result["H_raw"])          # e.g. 2.32
  print(result["H_improved"])     # e.g. 0.15
  print(result["delta_H"])        # e.g. 2.17
"""

MINIMAL_INSTRUCTION = "You are an AI agent that solves problems using tools."

# Core functions:

# 1. summarize_to_head(text, max_tokens=80, model)
#    - Summarize an arbitrary-length text into <= max_tokens tokens.
#    - Used to produce the 80-token Head for both H_raw and H_improved inputs.
#    - temperature = 0 (deterministic compression)
#    - Returns: summarized text (str)

# 2. trim_to_tail(text, max_tokens=150)
#    - Meaning-preserving trim: remove whitespace and filler only.
#    - Does NOT summarize or rewrite. Preserves key error logs and technical
#      keywords verbatim.
#    - Short inputs pass through unchanged.
#    - Used for the Tail in every ReAct step's context construction.

# 3. measure_semantic_entropy(input_context, model, n_samples=10)
#    - input_context = [minimal instruction] + [query or 300-token state]
#    - Fix input_context as the system prompt and sample the user question
#      n_samples times with temperature=0.7.
#    - Measurement question: "What concrete action will the agent take next?"
#    - Embed responses with Together AI's Llama-3 embedding model.
#    - Cluster on cosine similarity (AgglomerativeClustering, average linkage,
#      distance_threshold=0.15).
#    - Return Shannon entropy of the cluster distribution.

# 4. semantic_cluster(responses)
#    - Embed N responses via Together AI (base_url="https://api.together.xyz/v1")
#      using the OpenAI-compatible interface.
#    - Cluster: scikit-learn AgglomerativeClustering(
#          metric="cosine", linkage="average", distance_threshold=0.15
#      )
#    - Return: list of cluster labels.

# 5. inverse(raw_prompt, model, n_samples=10)
#    - 3-step independent prompt chaining (Prompt Chaining via Explicit Injection).
#    - Each step runs independently with no conversation history; the previous
#      step's output is explicitly injected into the next template.
#    - Step 2 performs Macro-level Backward Chaining: micro-steps ignored, only
#      decisive logical jumps derived. Stopping conditions:
#        (a) Logical threshold: derived condition can start with information
#            already in the initial request.
#        (b) Structural threshold: full backward chain ≤ 5 steps (k=4).
#    - Entropy is measured only at the entrance (raw_prompt) and the exit
#      (improved_prompt). Intermediate steps are not measured (different
#      question → different denominator → not comparable on same curve).
#    - temperature = 0 for all LLM calls inside inverse() (deterministic).
#
#    Pseudo-code:
#
#    def inverse(raw_prompt, model, n_samples=10):
#        # 1. Original query → 80-token summary → measure H_raw
#        raw_summary = summarize_to_head(raw_prompt, max_tokens=80, model=model)
#        H_raw = measure_semantic_entropy(
#            f"{MINIMAL_INSTRUCTION}\n\n{raw_summary}", model, n_samples
#        )
#
#        # 2. Run the 3 inverse-model steps (independent prompt chaining)
#        target = llm_call(prompt_1(raw_prompt), model, temperature=0)
#        inversion = llm_call(prompt_2(target, raw_prompt), model, temperature=0)
#        improved_prompt = llm_call(
#            prompt_3(raw_prompt, target, inversion), model, temperature=0
#        )
#
#        # 3. Refined query → 80-token summary → measure H_improved
#        improved_summary = summarize_to_head(improved_prompt, max_tokens=80, model=model)
#        H_improved = measure_semantic_entropy(
#            f"{MINIMAL_INSTRUCTION}\n\n{improved_summary}", model, n_samples
#        )
#
#        return {
#            "raw_prompt": raw_prompt,
#            "improved_prompt": improved_prompt,
#            "raw_summary": raw_summary,
#            "improved_summary": improved_summary,
#            "target": target,
#            "inversion": inversion,
#            "H_raw": H_raw,
#            "H_improved": H_improved,
#            "delta_H": H_raw - H_improved,
#            "total_tokens_used": total_tokens,
#            "pre_processing_time": time_taken,
#        }

# 6. detect_loop(entropy_history, H_raw, alpha=0.3, window=3)
#    - Detect loop regions in the entropy curve during agent execution.
#    - threshold = alpha × H_raw (based on the raw prompt's semantic entropy)
#    - Both A and B use H_raw as the reference (B does NOT use H_improved).
#    - Condition: d²H/dt² ≈ 0 (within window) AND H > alpha × H_raw
#    - Returns: {"is_loop": bool, "loop_start_step": int or None}
```

## Caching policy

Caching is **default-on**. The `--no-cache` flag disables it. Three caches:

| Cache | Key | Stored value | Reason |
|---|---|---|---|
| `cache/inverse/{key}.json` | `sha256(task_id + model + n_samples + cache_version)` | full `inverse()` return dict | inverse() is expensive (3 LLM calls + 2 entropy measurements); deterministic given the same task |
| `cache/h_raw/{key}.json` | `sha256(task_id + model + n_samples + cache_version)` | `{"H_raw": float, "raw_summary": str}` | Shared between conditions A and B; measured once per task |
| `cache/tavily/{key}.json` | `sha256(tool_name + query_or_url + cache_version)` | raw Tavily response JSON | Dominant cost saver across reruns |

**H_improved is deliberately NOT cached**: this is the sanity check for inverse cache integrity. If the inverse cache is corrupted or stale, the freshly recomputed `H_improved` will disagree with the cached `inverse_result["H_improved"]`, surfacing the problem immediately.

**Cache invalidation**: bump `CACHE_VERSION` (top of `benchmark.py`) to invalidate everything. Changing model, n_samples, or the distance_threshold should also bump the version.

**Cache helper functions**:

```python
import hashlib, json
from pathlib import Path

CACHE_VERSION = "v2.7.9-001"
CACHE_ROOT = Path("cache")

def _cache_key(*parts) -> str:
    """Build a deterministic cache key from arbitrary parts."""
    payload = "||".join(str(p) for p in parts) + f"||v={CACHE_VERSION}"
    return hashlib.sha256(payload.encode()).hexdigest()[:32]

def cache_get(subdir: str, key: str):
    """Return cached value or None if miss. subdir ∈ {'inverse', 'h_raw', 'tavily'}."""
    path = CACHE_ROOT / subdir / f"{key}.json"
    if not path.exists():
        return None
    return json.loads(path.read_text())

def cache_set(subdir: str, key: str, value) -> None:
    """Store value. Caller decides contents (must be JSON-serializable)."""
    path = CACHE_ROOT / subdir / f"{key}.json"
    path.parent.mkdir(parents=True, exist_ok=True)
    path.write_text(json.dumps(value))

def cache_hit(subdir: str, key: str) -> bool:
    return (CACHE_ROOT / subdir / f"{key}.json").exists()
```

## Failure handling

| Failure | Handling |
|---|---|
| Tavily timeout/error | Place the error message as the Observation; let the agent decide (retry or switch tools). No new termination category. |
| LLM rate limit | Exponential backoff at the API call layer. Does not affect `terminated_by`. |
| LLM empty response (rare) | Retry once with the same prompt. If still empty, `terminated_by = "error"`. |
| `inverse()` internal failure | Retry once. If still failing, `terminated_by = "error"` and `verifier_passed = "N/A"`. |
| Agent returns text without any tool call | Retry the same step once. If still no tool call, force a `final_answer` call with the agent's best-effort answer. |

**The `error` termination value**:
- Never counted as a loop.
- Never counted as completion.
- Tasks terminated with `error` are reported in a separate run-summary category.
- If the error rate exceeds 10% in a run, that run is excluded from analysis and the cause is investigated.

## How this differs from prior work

| Prior work | What this project adds |
|------------|------------------------|
| **Wolpert & Kawato (1998)** — forward/inverse model theory for motor control | Applies the cognitive control model to **prompt refinement for LLM agents** |
| **Correa & de Matos (2025)** — *Entropy-Guided Loop*: an inference-time refinement loop triggered by token-level uncertainty | Removes entropy **before execution (pre-processing)**, not during execution |
| **Pandita et al. (2025)** — *ProRefine* (NeurIPS): an agentic loop that refines prompts at inference time using textual feedback | Quantifies the effect with **the entrance/exit entropy difference (ΔH)** |
| **Xu et al. (2025)** — *EPO*: entropy control during multi-turn LLM agent RL training | Measures **ΔH at inference time**, not during training |
| **Kuhn et al. (2023)** — *Semantic Uncertainty* (ICLR Spotlight): entropy measurement based on semantic equivalence | Re-purposes SE as **a metric for the effect of pre-processing (ΔH)** |

Nobody has put these three together:
1. Apply the inverse model from motor neuroscience to prompt refinement.
2. Measure the semantic entropy of the raw and improved prompts using the same question.
3. Test, A/B, whether pre-processing reduces the agent loop rate on a standardized benchmark (GAIA).

## v0 → v1 expansion plan

Depending on v0 results:

- **It works**: add a forward model (verify against the Target during execution) → run condition C (inverse + forward).
- **It doesn't work**: add entropy measurement at the intermediate steps → diagnose where information is lost (Target or Invert).
- **Partial effect**: per-level analysis (Level 1 vs 2 vs 3), per-question-type analysis → specialize the inverse-model system prompt by task category.
- **GAIA H_raw distribution too low**: if GAIA's questions are too clear to show an inverse model effect, introduce an ambiguation step in v1 (self-authored ambiguous task set with LLM-judge verifier as a sub-result).

## Appendix A — Known issue: GAIA temporal drift

GAIA was created in 2023. Some questions implicitly reference "the current moment" (e.g., "Who is the current CEO of X?"). For these:

- Tavily will return 2026-era search results.
- The ground truth in `Final answer` reflects 2023 facts.
- The agent may answer correctly for the present day but be marked incorrect by the verifier.

This is a known issue with GAIA itself, not addressable in the v0 spec. During result analysis:

1. Manually review tasks where `verifier_passed == False` to identify "temporal mismatch" cases.
2. Report temporal-mismatch tasks in a separate analysis category.
3. The primary correctness-rate analysis should exclude temporal-mismatch tasks.

This is a post-run analysis rule, not automated. A v1 consideration is switching to a GAIA refresh (if available) or introducing a self-authored task set that is time-invariant.
